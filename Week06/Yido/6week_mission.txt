### 아키텍처의 필요성

여러분이 처음 Android 앱을 만들 때는 아마 이런 코드를 작성 하셨을 겁니다.

```kotlin
class MainActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // 버튼 클릭 이벤트
        val button = findViewById<Button>(R.id.btn_load)
        button.setOnClickListener {
            // 직접 네트워크 통신
            val users = fetchUsersFromServer()  // ❌ UI 레이어에서 네트워크 호출
            
            // 직접 DB 접근
            val db = Room.databaseBuilder(this, AppDatabase::class.java, "db").build()
            db.userDao().insertAll(users)       // ❌ UI 레이어에서 DB 접근
            
            // UI 업데이트
            val textView = findViewById<TextView>(R.id.tv_result)
            textView.text = users.toString()
        }
    }
}
```

이 코드의 문제점은 다음과 같습니다.

- **테스트 불가** : Activity는 Android 기기 없이 테스트할 수 없습니다.
- **유지보수 어려움 :** 코드가 길어질수록 어디를 수정해야 할지 찾기 어렵습니다.
- **화면 회전 문제 :** 화면을 회전하면 네트워크 요청이 처음부터 다시 시작됩니다.

이러한 문제를 해결하기 위해서 아키텍처를 설계하여 코드의 품질을 높여야 합니다! MVVM, MVI, MVC, MVVM 등 다양한 아키텍처 방식이 존재하나, 해당 워크북에서는 MVVM만 다루도록 하겠습니다.

### MVVM이란 무엇인가?

**MVVM**은 **M**odel - **V**iew - **V**iew**M**odel의 약자입니다.

MVVM의 핵심 원칙은 바로 **단방향 데이터 흐름** 입니다. 데이터를 한 방향으로 흐르게 만들어 개발자가 **코드를 예측 가능하도록** 하는 것이 MVVM 패턴에 핵심입니다.

그림을 통해 MVVM 아키텍처의 구조는 어떤 것인지 확인하겠습니다.

![image.png](attachment:69e5e2b8-aaa3-46a0-9130-5be763e995c8:image.png)

#### View (뷰)

View는 사용자에게 보이는 화면을 담당합니다. 대표적인 예시로 Activity, Fragment가 되겠습니다. (Compose에서는 Screen) View에서 사용자에게 보이는 화면을 제외하고 담당하는 업무가 2가지 있습니다.

- 사용자 입력을 받아서 ViewModel에게 전달
- ViewModel의 데이터 변화를 화면에 반영

아래 코드는 MVVM 패턴에서의 View의 역할을 올바르게 표현한 코드입니다.

```kotlin
// ✅ 좋은 View 코드
class TestFragment : Fragment() {
    
    private val viewModel: TestViewModel by viewModels()
    
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        super.onViewCreated(view, savedInstanceState)
        
        // ViewModel의 데이터를 관찰하여 화면에 반영
        // 관찰하는 방법은 아래에 기술하겠습니다.
        viewModel.users.observe(viewLifecycleOwner) { users ->
            adapter.submitList(users)  // 화면 업데이트만
        }
        
        // 버튼 클릭은 ViewModel에게 위임
        binding.btnRefresh.setOnClickListener {
            viewModel.refreshUsers()  // 직접 처리하지 않고 ViewModel에 맡김
        }
    }
}
```

#### ViewModel (뷰모델)

ViewModel은 View와 Model 사이의 중간 관리자입니다. 비즈니스 로직을 담당하여 데이터를 가공하고, 보관하며 View의 이벤트를 받아 Model에 요청하거나, Model에서 가져온 데이터를 View에 건네주는 역할입니다.

```kotlin
// ✅ 좋은 ViewModel 코드
class TestViewModel(
		// Respository pattern (아래에 추가적으로 기술되어 있습니다)
    private val userRepository: UserRepository  // Model(Repository)를 주입받음
) : ViewModel() {
    
    // View가 관찰할 수 있는 데이터
    private val _users = MutableLiveData<List<User>>()
    val users: LiveData<List<User>> = _users
    
    // View로부터 이벤트를 받는 함수
    fun refreshUsers() {
        viewModelScope.launch {
            val result = userRepository.getUsers()
            _users.value = result
        }
    }
}
```

<aside>
💡

여기서 중요한 점은 ViewModel에서 `Context`  또는 View의 참조를 가지고 있으면 안됩니다!

ViewModel 의 생명주기는 Fragment가 아닌 다른 방식으로 적용되어 있어 crash 오류가 발생할 수 있으며, 단방향 원칙이 깨지게 됩니다!

</aside>

#### Model (모델)

실제 데이터를 다루는 레이어입니다. Model 영역에는 아래 내용이 포함되어 있습니다.

- 데이터의 형태를 정의하는 **Data Class**
- 데이터를 가져오고 저장하는 **Repository**
- 로컬 DB (Room), 원격 API (Retrofit) 등

```kotlin
// Data Class - 데이터의 형태 정의
data class User(
    val id: Int,
    val name: String,
    val email: String
)

// Repository - 데이터 소스 추상화
class UserRepository(
    private val userApi: UserApi,  // 원격 API
    private val userDao: UserDao   // 로컬 DB
) {
    suspend fun getUsers(): List<User> {
        return try {
            val remoteUsers = userApi.fetchUsers()  // 네트워크에서 가져오기
            userDao.insertAll(remoteUsers)          // 로컬 DB에 저장
            remoteUsers
        } catch (e: Exception) {
            userDao.getAllUsers()
        }
    }
}
```

이렇게 전체적인 MVVM 구조를 살펴보았습니다. View와 Model은 지금까지 미션을 수행하시면서 충분히 이해가 되실 것으로 생각합니다. 그러나 ViewModel은 해당 내용 외에도 추가적인 개념이 많이 존재하기에 이를 자세히 다뤄보겠습니다.

#### LiveData와 StateFlow

ViewModel에서 View로 데이터를 전달하는 두 가지 방법입니다.

- **LiveData (전통적인 방법)**

앞서 ViewModel에서 관찰할 수 있는 데이터를 생성할 때 사용한 방식입니다.

```kotlin
class TestViewModel : ViewModel() {
    
    // private: ViewModel 내부에서만 수정 가능
    private val _userName = MutableLiveData<String>()
    
    // public: View에서는 읽기만 가능
    val userName: LiveData<String> = _userName
    
    fun updateName(name: String) {
        _userName.value = name           // 메인 스레드에서
    }
}

// View에서 관찰하기
viewModel.userName.observe(this) { name ->
    binding.tvName.text = name
}
```

- **StateFlow (최신 방법, Coroutines와 함께)**

```kotlin
class UserViewModel : ViewModel() {
    
    private val _uiState = MutableStateFlow(UserUiState())
    val uiState: StateFlow<UserUiState> = _uiState.asStateFlow()
    
    fun loadUser(userId: Int) {
        viewModelScope.launch {
            val user = repository.getUser(userId)
            
            _uiState.update { 
                it.copy(
                    user = user
                )
            }
        }
    }
}

// UI 상태를 하나의 데이터 클래스로 관리하는 패턴
data class UserUiState(
    val isLoading: Boolean = false,
    val user: User? = null,
    val error: String? = null
)

// View에서 수집하기
lifecycleScope.launch {
    viewModel.uiState.collect { state ->
        state.user?.let { binding.tvName.text = it.name }
    }
}
```

**LiveData vs StateFlow 비교**

| 특징 | LiveData | StateFlow |
| --- | --- | --- |
| 생명주기 인식 | ✅ 자동 | ❌ 수동 처리 필요 |
| 초기값 필요 | ❌ | ✅ 필요 |
| 코루틴 친화성 | 보통 | 매우 좋음 |
| Google 공식 추천 | 예전 표준 | 현재 표준 |

둘의 가장 큰 차이점은 생명주기 인식 방법 입니다. `LiveData`는 자동이지만, `StateFlow`는 수동으로 생명주기와 연결시켜야 합니다. 

그러나 서버와 연결하여 데이터를 업데이트 하는 방식은 비동기 처리로 진행되어 Coroutine을 사용하기 때문에 이와 친화적인 `StateFlow`를 최근에는 많이 선호하고 있습니다. 이번 워크북 미션도 `StateFlow`를 이용하시면 좋을 것 같습니다.

#### 1. Hilt 설정하기

Hilt는 **Google에서 권장하는 Android 전용 의존성 주입(DI) 라이브러리** 입니다. 쉽게 ViewModel이 Repository를 사용하기 위해서 객체를 자동으로 만들어서 넣어주는 도구라고 생각하시면 됩니다.

Hilt를 쓰려면 먼저 `build.gradle`에 의존성을 추가하고, `Application` 클래스를 하나 만들어야 합니다.

```kotlin
// 해당 예제에서는 이해를 돕기 위해 gradle에 바로 작성하였으나
```

#### 2. Model 만들기

데이터의 형태를 정의하는 단계입니다.

```kotlin
// data/model/Todo.kt
@Entity(tableName = "todos")
data class Todo(
    @PrimaryKey(autoGenerate = true)
    val id: Int = 0,
    val title: String,
    val isCompleted: Boolean = false,
    val createdAt: Long = System.currentTimeMillis()

```

<aside>
💡

**왜 Model을 먼저 만드는가?**

프로젝트를 시작하게 되었을 때 이상적인 상황은 백엔드의 작업이 완료된 후 프론트가 개발을 시작하는 상황입니다. 그러나 현실적인 조건에 의해 보통은 백엔드와 프론트엔드 동시에 작업을 시작하게 됩니다. 작업 속도를 따졌을 때 보통은 프론트가 먼저 끝나기 때문에 필요한 데이터를 서버 개발과 관계없이 미리 필요한 데이터를 정의하여 활용하기 위함입니다.

#### 3. Repository 만들기

이 단계에서 Repository가 처음 등장합니다. 바로 코드를 작성하기 전에, "왜 필요한지"부터 짚고 넘어가겠습니다.

#### 3-1. Repository가 없으면 어떻게 될까?

```kotlin
// ❌ Repository 없는 ViewModel — 모든 걸 혼자 떠안고 있음
class BadViewModel : ViewModel() {
    private val db = Room.databaseBuilder(...)  // ViewModel이 DB를 직접 만든다
        .build()

    fun loadTodos() {
        viewModelScope.launch {
            val todos = db.todoDao().getAll()   // DB 로직이 ViewModel에
            _todos.value = todos
        }
    }
}
```

위 코드의 문제점은 다음과 같습니다.

- **책임 과부화** : ViewModel이 데이터 저장·조회·오류처리까지 모두 담당
- **재사용 불가 :** 다른 화면에서 Todo를 쓰려면 같은 DB 코드를 복사해야 함
- **테스트 불가 :** 실제 DB 없이는 ViewModel 테스트 자체가 불가

> 
> 
> 
> 예시)
> 
> 도서관을 예시로 들겠습니다.
> 
> - **Repository(사서)** — 책이 어느 서가에 있는지, 디지털 자료인지, 외부 도서관에서 빌려야 하는지 모두 알고 있는 사람
> - **ViewModel(독자)** — 사서에게 "이 책 주세요"라고만 하면 됨. 어디서 어떻게 가져오는지는 몰라도 됨
> - **Room / API(서가·디지털 자료실)** — 실제 데이터가 저장된 곳
> 
> Repository 없이는 독자가 서가를 직접 뒤져야 합니다…
> 

#### 3-2. Repository 역할 정리

```
ViewModel ─── "Todo 목록 줘" ───► Repository
                                      │
                     ┌────────────────┤ 데이터 출처 결정
                     │                │
                     ▼                ▼
              Room (로컬 DB)    Retrofit (원격 API)
              오프라인 지원      최신 데이터
```

Repository가 하는 일:

- **데이터 출처 추상화** : ViewModel은 DB인지 API인지 알 필요 없음
- **캐싱 전략** : 언제 DB를 쓰고 언제 네트워크를 쓸지 결정
- **에러 처리 통합 :** 오류를 한 곳에서 일관되게 처리

#### 3-3. Interface로 Repository 추상화하기

실전에서는 Repository를 **Interface** 로 먼저 정의하고 구현체를 따로 만듭니다. 

ViewModel은 `TodoRepository`(계약서)만 보고 코드를 작성합니다. 그래서 실제 앱에서는 `TodoRepositoryImpl`을, 테스트할 때 ViewModel 코드 수정 없이 더미 Repository로 쉽게 교체할 수 있기 때문입니다.

```kotlin
// domain/repository/TodoRepository.kt  ← Interface: "이런 기능을 제공하겠다"는 계약서
interface TodoRepository {
    fun getTodosStream(): Flow<List<Todo>>
    suspend fun insert(todo: Todo)
    suspend fun update(todo: Todo)
    suspend fun delete(todo: Todo)
}
```

```kotlin
// data/repository/TodoRepositoryImpl.kt  ← 실제 구현체
class TodoRepositoryImpl @Inject constructor(  // ← Hilt가 TodoDao를 자동으로 주입
    private val todoDao: TodoDao               // 반드시 @Inject가 붙어야 합니다!  
) : TodoRepository {
    override fun getTodosStream(): Flow<List<Todo>> = todoDao.getAllTodosFlow()
    override suspend fun insert(todo: Todo) = todoDao.insert(todo)
    override suspend fun update(todo: Todo) = todoDao.update(todo)
    override suspend fun delete(todo: Todo) = todoDao.delete(todo)
}
```

#### 3-4. Hilt Module 작성

Room DB와 Repository Interface 연결은 **Hilt가 직접 만들 수 없는 영역**이라 Module에서 "이렇게 만들어!"라고 설명서를 써줍니다.

```kotlin
// di/DatabaseModule.kt
@Module
@InstallIn(SingletonComponent::class)  // 앱 전체에서 하나만 유지
object DatabaseModule {

    @Provides
    @Singleton  // ← DB는 무조건 하나만! 없으면 호출마다 새로 생성됨
    fun provideTodoDatabase(@ApplicationContext context: Context): TodoDatabase {
        return Room.databaseBuilder(context, TodoDatabase::class.java, "todo_db").build()
    }

    @Provides
    fun provideTodoDao(database: TodoDatabase): TodoDao = database.todoDao()
}
```

```kotlin
// di/RepositoryModule.kt
@Module
@InstallIn(SingletonComponent::class)
abstract class RepositoryModule {

    // "TodoRepository 달라고 하면 TodoRepositoryImpl을 줘라"
    @Binds
    @Singleton
    abstract fun bindTodoRepository(impl: TodoRepositoryImpl): TodoRepository
}
```

> **`@Binds`가 하는 일**
Interface(`TodoRepository`)와 구현체(`TodoRepositoryImpl`)를 연결해주는 역할입니다.
이제 ViewModel이 `TodoRepository`를 요청하면 Hilt가 `TodoRepositoryImpl`을 자동으로 주입해줍니다.
>

#### 4. ViewModel 만들기 — `@HiltViewModel`

`@HiltViewModel` + `@Inject constructor`를 붙이면, Hilt가 Repository 주입과 ViewModel 생성을 모두 자동으로 처리합니다.

```kotlin
// ui/main/MainViewModel.kt
@HiltViewModel
class MainViewModel @Inject constructor(
    private val repository: TodoRepository  // ← Interface 타입! 구현체가 뭔지 몰라도 됨
) : ViewModel() {

    private val _uiState = MutableStateFlow(TodoUiState())
    val uiState: StateFlow<TodoUiState> = _uiState.asStateFlow()

    init {
        // 앱 시작 시 DB 변화를 구독
        viewModelScope.launch {
            repository.getTodosStream().collect { todos ->
                _uiState.update { it.copy(todos = todos) }
            }
        }
    }

    fun addTodo(title: String) {
        if (title.isBlank()) return
        viewModelScope.launch { 
		        repository.insert(
				        Todo(title = title)
				    )
				}
    }

    fun toggleComplete(todo: Todo) {
        viewModelScope.launch {
		        repository.update(
				        todo.copy(isCompleted = !todo.isCompleted)
				    )
				}
    }

    fun deleteTodo(todo: Todo) {
        viewModelScope.launch { 
		        repository.delete(todo) 
		    }
    }
}

// UI 상태를 하나의 데이터 클래스로 관리
data class TodoUiState(
    val todos: List<Todo> = emptyList()
)
```

#### 5. View 만들기 — `@AndroidEntryPoint`

Activity에 `@AndroidEntryPoint`만 붙이면, `by viewModels()` 한 줄로 ViewModel 주입이 완료됩니다.

```kotlin
// ui/main/MainActivity.kt
@AndroidEntryPoint
class MainActivity : AppCompatActivity() {

    // ✅ Factory 없이 이 한 줄로 끝!
    private val viewModel: MainViewModel by viewModels()

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        binding = ActivityMainBinding.inflate(layoutInflater)
        setContentView(binding.root)

        lifecycleScope.launch {
            viewModel.todos.collect { todos ->
                adapter.submitList(todos)
                binding.tvEmpty.isVisible = todos.isEmpty()
            }
        }

        binding.btnAdd.setOnClickListener {
            viewModel.addTodo(binding.etTodo.text.toString())
            binding.etTodo.text?.clear()
        }
    }
}
```

#### 5-1. Data Binding (XML 한정)

지금껏 xml에 변경되는 UI를 반영하기 위해서는 View에 존재하는 Fragment, Activity에서 isVisible , text등 함수를 통해 변경하였습니다. 그러나 Data Binding을 사용하면 XML에서 직접 ViewModel의 데이터를 반영하여 업데이트가 가능합니다.

```kotlin
// build.gradle (app)
android {
    buildFeatures {
        dataBinding = true
    }
}
```

```kotlin
<!-- activity_main.xml -->
<layout xmlns:android="http://schemas.android.com/apk/res/android">
    
    <data>
        <variable
            name="vm"
            type="com.example.app.ui.MainViewModel" />
    </data>
    
    <LinearLayout ...>
        
        <!-- ViewModel의 데이터를 직접 바인딩 -->
        <TextView
            android:text="@{vm.todos.first().title}"
            android:visibility="@{vm.todos.first().isEmpty ? View.GONE : View.VISIBLE}" />
        
        <!-- 버튼 클릭도 바인딩 가능 -->
        <Button
            android:onClick="@{() -> vm.addTodo()}" />
            
    </LinearLayout>
</layout>
```

위 코드처럼 반드시 데이터 바인딩을 사용할 때 `@{ }` 방식으로 감싸줘야 한다는 것을 기억해주세요!
